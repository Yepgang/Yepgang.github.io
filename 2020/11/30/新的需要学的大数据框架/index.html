hbase

  • backup
    • 性质：分布式列族数据库，建立在HDFS上，用于处理超大型数据集并且需要常量级随机读写操作
    • 优势
      • 普通数据库：通常利用复制和分区来扩展数据库，但是这些是后天的设计，导致数据库很难运行和维持，当进行原有的操作比如：连接、复杂查询、触发器、视图等，拓展的RDBMS的运行时间难以接受
      • Hbase：专门设计用来处理海量数据，通过增加节点可以实现数据库规模的线性增长(当然不支持sql笑)
  • data model
    • e.g.
    • 基本概念
      • rowkey：类似逐渐，byte数组，按byte-order排序
      • 列族
        • 包括两部分——prefix：qualifier
        • 在数据处理的过程中可以动态添加qualifier，但是prefix需要提前定义(prefix其实就是列族)
        • 在底层存储中，列族成员内容存在同一个文件系统，因此hbase是基于列族存储的
      • cells、versions(…)每个cell保存多个versions的数据，由时间戳进行索引,byte数组，底层存储结构
    • region：水平划分，由连续的行构成，不同的区域存放在不同的节点
  • 具体实现
    • region server:管理分区
    • master:分配并管理regionServer，监控每个regionServer的状态、负载均衡和故障转移
    • zookeeper
      • 协调master和server的工作，记录hbase元数据的目录位置，meta中含有master/server的位置，这也是client读写的第一步
      • 参与master分配server的过程，解决分配的过程出现的问题
    • hdfs：底层存储
    • 深入理解读写过程
      • region server的结构
      • WAL：全称Write Ahead Log， 属于分布式系统上的文件。主要用来存储还未被持久化到磁盘的新数据。如果新数据还未持久化，节点发生宕机，那么就可以用WAL来恢复这些数据
      • BlockCache：是一个读缓存。它存储了被高频访问的数据。当这个缓存满了后，会清除最近最少访问的数据
      • MemStore: 是一个写缓存。它存储了还未被写入磁盘的数据。它会在写入磁盘前，对自身数据进行排序，从而保证数据的顺序写入。每个region的每个colum family对应唯一的memstore
      • HFiles：按照键值字典序存储数据
      • 写：WAL->MemStore->刷新->Hfile;读：BlockCache->MemStore->Hfile->返回所有版本合并后的数据

zooKeeper

  • 本质：为分布式应用提供协调服务。在分布式任务中，不同的节点存在信息差，所谓协调服务就是要解决这个问题，zookeeper提供一系列工具处理分布式应用中的partial failures
  • 特点
    • 简单的文件系统，带有最基本的功能比如排序、通知
    • 功能强大，提供众多的的协调服务，比如构建分布式队列、选举leader
    • 高可用用：可以解决分布式程序中的单点故障问题，每个server保存一份相同的数据副本
    • 交互中转站：可以作为不同节点交互的中转站
    • 性能惊人：ZooKeeper集群的吞吐量基准是每秒超过10000次操作
    • 服务器集群：一个leader，多个follwer；只要有超过一办的服务器还在运行，zooKeeper就能正常运行
  • 工作机制
    • 基于观察者模式，接受服务器的注册和下线，维持当前服务器列表并且与客户端交互
    • zookeeper数据模型可以看成一颗树，有统一的节点概念znode，每个znode由路径唯一标识
    • 运行过程中只要有超过一半的节点正常运行zookeeper就可以正常运行，节点分为leader和foller
  • 应用场景：
    • 统一命名
    • 统一配置
    • 统一集群管理
    • 服务器动态上下线/软负载均衡
  • 安装配置分布式zooKeeper,再次熟悉了xsync脚本
    • zoo.cfg/myid
  • 服务器选举模式：
    • conf包含服务器和客户端的启动脚本
  • 节点类型(临时？序列？)
  • 客户端shell指令
  • 监听是zookeeper最核心的功能
