版本控制(git)

Posted on 2020-12-24 | In 工程能力

前言

版本控制系统 (VCSs) 是一类用于追踪源代码（或其他文件、文件夹）改动的工具,这些工具可以帮助我们管理代码的修改历史;让协作编码变得更方便

Git的接口有些丑陋，但是它的底层设计和思想却是非常优雅的。丑陋的接口只能靠记硬背，而优雅的底层设计则非常容易被人理解。因此，我们将通过一种自底向上的式向您介绍 Git。我们会从数据模型开始，最后再学习它的接口。一旦您搞懂了 Git的数据模型，再学习其接口并理解这些接口是如何操作数据模型的就非常容易了。

分布式版本控制系统：客户端并不只提取最新版本的文件快照， 而是把代码仓库完整地镜像下来，包括完整的历史记录。这么一来，任何一处协同工作用的服务器发生故障，事后都可以用任何一个镜像出来的本地仓库恢复。 因为每一次的克隆操作，实际上都是一次对代码仓库的完整备份

从概念上来说，其它大部分版本控制系统以文件变更列表的方式存储信息；Git 不按照以上方式对待或保存数据。反之，Git 更像是把数据看作是对小型文件系统的一系列快照。在 Git中，每当你提交更新或保存项目状态时，它基本上就会对当时的全部文件创建一个快照并保存这个快照的索引。为了效率，如果文件没有修改，Git 不再重新存储该文件，而是只保留一个链接指向之前存储的文件。Git 对待数据更像是一个快照流。

在 Git 中的绝大多数操作都只需要访问本地文件和资源，一般不需要来自网络上其它计算机的信息

Git中所有的数据在存储前都计算校验和，然后以校验和来引用。Git 用以计算校验和的机制叫做 SHA-1 散列。这是一个由40个十六进制字符（0-9 和 a-f）组成的字符串，基于Git中文件的内容或目录结构计算出来，Git 数据库中保存的信息都是以文件内
容的哈希值来索引，而不是文件名

你执行的 Git 操作，几乎只往 Git 数据库中 添加 数据。 你很难使用 Git 从数据库中删除数据，也就是说 Git 几乎不会执行任何可能导致文件不可恢复的操作

git基础

  • 基本概念
    • 文件：blob对象
    • 目录：tree
    • 快照：被追踪的最顶层的树，在git中被称为提交commit
    • 历史记录：快照组成的有向无环图
    • blob、tree、commit都是对象，基于SHA-1 hash寻址
    • 引用：指向commit的指针，方便将hash值可读化
      • master通常指向主分支最新一次的提交
      • HEAD指向当前位置
    • 对象和引用构成git仓库，git指令对应对commit tree的操作
  • Git项目拥有三个阶段：工作区、暂存区以及 Git目录
    • 工作区是对项目的某个版本独立提取出来的内容。这些从 Git 仓库的压缩数据库中提取出来的文件，放在磁盘上供你使用或修改
    • 暂存区：staging area，指定下一次提交的快照中包括哪些改动
    • Git仓库目录是 Git用来保存项目的元数据和对象数据库的地方。这是Git中最重要的部分，从其它计算机克隆仓库时，复制的就是这里的数据

    • (RX1|RX2) 任何能够匹配RX1 或 RX2的结果
    • ^ 行首
    • $ 行尾
    • \d \w
    • ()表示capture
  • 正则表达式适用的场景很多，包括grep

Shell工具和脚本

Posted on 2020-12-19 | In 工程能力

大多数shell都有自己的一套脚本语言，包括变量、控制流和自己的语法。shell脚本与其他脚本语言不同之处在于，shell脚本针对shell所从事的相关工作进行来优化。因此，创建命令流程（pipelines）、将结果保存到文件、从标准输入中读取输入，这些都是shell脚本中的原生操作，这让它比通用的脚本语言更易用。本节中，我们会专注于bash脚本，因为它最流行，应用更为广泛。

关于bash

  • 在shell脚本中使用空格会起到分割参数的作用，有时候可能会造成混淆
  • 以’定义的字符串为原义字符串，其中的变量不会被转义，而 “定义的字符串会将变量值进行替换，默认为”
  • bash使用了很多特殊变量来表示参数、错误代码和相关变量
    • $0 脚本名
    • $1 到 $9 - 脚本到参数。 $1 是第一个参数，依此类推。
    • $@ - 所有参数
    • $# - 参数个数
    • $? - 前一个命令的返回值
    • $$ - 当前脚本的进程识别码
    • !! - 完整的上一条命令，包括参数。常见应用：当你因为权限不足执行命令失败时，可以使用 sudo !!再尝试一次。
    • $_ - 上一条命令的最后一个参数。如果你正在使用的是交互式shell，你可以通过按下 Esc 之后键入 . 来获取这个值。
  • 返回值0表示正常执行，其他所有非0的返回值都表示有错误发生，可以搭配&&和||使用。
  • 同一行的多个命令可以用 ; 分隔
  • command substitution：$(cmd)
  • 在bash中进行比较时，尽量使用双方括号 [[ ]] 而不是单方括号 [ ]，这样会降低犯错的几率，尽管这样并不能兼容sh
  • ? 和 * 来匹配一个或任意个字符
  • 花括号{}:当你有一系列的指令，其中包含一段公共子串时，可以用花括号来自动展开这些命令
    • mv *{.py,.sh} folder(会移动所有 *.py 和 *.sh 文件)
    • touch {foo,bar}/{a..h}(创建foo/a, foo/b, … foo/h, bar/a, bar/b, … bar/h这些文件)
  • 函数只能用与shell使用相同的语言，脚本可以使用任意语言(脚本第一行定义命令解释器)
  • 函数会在当前的shell环境中执行，脚本会在单独的进程中执行。因此，函数可以对环境变量进行更改，比如改变当前工作目录，脚本则不行
  • tldr:simple man tec

文件描述符

Linux系统预留可三个文件描述符：0、1和2，他们的意义如下所示：

  • 0——标准输入（stdin）
  • 1——标准输出（stdout）
  • 2——标准错误（stderr）
    • grep foobar $file > /dev/null 2> /dev/null(将标准输出流和标准错误流重定向到Null)
    • ls e.txt 2>g.txt

文件夹导航

  • alias
  • fasd:在内部维护一个表，记录着最近操作的每个文件/目录的路径，以及他们的权重。这个权重会随着你操作这个目录的次数而增加，所以在使用关键字匹配路径时，最常用的路径就会被选中
  • 内置别名
    • alias a=’fasd -a’ # any
    • alias s=’fasd -si’ # show / search / select
    • alias d=’fasd -d’ # directory
    • alias f=’fasd -f’ # file
    • alias sd=’fasd -sid’ # interactive directory selection
    • alias sf=’fasd -sif’ # interactive file selection
    • alias z=’fasd_cd -d’ # cd, same functionality as j in autojump
    • alias zz=’fasd_cd -d -i’ # cd with interactive selection
  • 使用方法
    • a foo 列出最近操作的路径匹配’foo’的文件与目录
    • f foo 列出最近操作的路径匹配’foo’的文件
    • f foo -e vim 打开…
    • d foo 列出最近操作的路径匹配’foo’的目录
    • s foo 列出最近操作的路径匹配’foo’的文件与目录，并可以通过序号选择
    • sf foo 列出最近操作的路径匹配’foo’的文件，并可以通过序号选择
    • sd foo 列出最近操作的路径匹配’foo’的目录，并可以通过序号选择
    • z foo cd到最近操作的匹配’foo’并且得分最高的目录
    • zz foo 列出最近操作的路径匹配’foo’的目录，通过序号选择，然后cd进目录

关于shell

  • Shell俗称壳（用来区别于核），命令解析器，接收用户命令，调用相应的应用程序(怎么知道调用对应的程序??)
  • 常见的shell有bash、dash…dash主要目的是执行脚本而不是为了交互，速度快但是功能少于bash
  • #!/bin/sh:一般见于shell文件的第一行，用于指定解释该shell文件的sh路径
  • 如今Debian和Ubuntu中，/bin/sh默认已经指向dash，可以通过指令ls -l /bin/sh确认，通过sudo dpkg-reconfigure dash可以修改sh为bash，同理可切换bash为dash
  • /etc/shells包含支持的shell
  • 一个有意思的指令：echo 3 | sudo tee brightness

  • 可以用echo和重定向符写一些简单的文件(平时用vim习惯了)

    • a：访问时间/m：修改时间/c：如果该文件不存在则不创建
  • grep
    • 用于查找内容包含指定的范本样式的文件,如果不指定查找文件或者目录则从标准输入设备读取数据，找到返回值为0，找不到返回值为1
    • i:忽略大小写；C：上下文；v:反选
    • grep test *file
    • grep -r update /etc/acpi
  • chmod
    • User、Group、Other users对应ugo
    • rwx对应421
    • chmod ug+w,o-w file1.txt file2.txt
    • chmod a=rwx file
    • chmod 777 file

hbase

  • backup
    • 性质：分布式列族数据库，建立在HDFS上，用于处理超大型数据集并且需要常量级随机读写操作
    • 优势
      • 普通数据库：通常利用复制和分区来扩展数据库，但是这些是后天的设计，导致数据库很难运行和维持，当进行原有的操作比如：连接、复杂查询、触发器、视图等，拓展的RDBMS的运行时间难以接受
      • Hbase：专门设计用来处理海量数据，通过增加节点可以实现数据库规模的线性增长(当然不支持sql笑)
  • data model
    • e.g.
    • 基本概念
      • rowkey：类似逐渐，byte数组，按byte-order排序
      • 列族
        • 包括两部分——prefix：qualifier
        • 在数据处理的过程中可以动态添加qualifier，但是prefix需要提前定义(prefix其实就是列族)
        • 在底层存储中，列族成员内容存在同一个文件系统，因此hbase是基于列族存储的
      • cells、versions(…)每个cell保存多个versions的数据，由时间戳进行索引,byte数组，底层存储结构
    • region：水平划分，由连续的行构成，不同的区域存放在不同的节点
  • 具体实现
    • region server:管理分区
    • master:分配并管理regionServer，监控每个regionServer的状态、负载均衡和故障转移
    • zookeeper
      • 协调master和server的工作，记录hbase元数据的目录位置，meta中含有master/server的位置，这也是client读写的第一步
      • 参与master分配server的过程，解决分配的过程出现的问题
    • hdfs：底层存储
    • 深入理解读写过程
      • region server的结构
      • WAL：全称Write Ahead Log， 属于分布式系统上的文件。主要用来存储还未被持久化到磁盘的新数据。如果新数据还未持久化，节点发生宕机，那么就可以用WAL来恢复这些数据
      • BlockCache：是一个读缓存。它存储了被高频访问的数据。当这个缓存满了后，会清除最近最少访问的数据
      • MemStore: 是一个写缓存。它存储了还未被写入磁盘的数据。它会在写入磁盘前，对自身数据进行排序，从而保证数据的顺序写入。每个region的每个colum family对应唯一的memstore
      • HFiles：按照键值字典序存储数据
      • 写：WAL->MemStore->刷新->Hfile;读：BlockCache->MemStore->Hfile->返回所有版本合并后的数据

zooKeeper

  • 本质：为分布式应用提供协调服务。在分布式任务中，不同的节点存在信息差，所谓协调服务就是要解决这个问题，zookeeper提供一系列工具处理分布式应用中的partial failures
  • 特点
    • 简单的文件系统，带有最基本的功能比如排序、通知
    • 功能强大，提供众多的的协调服务，比如构建分布式队列、选举leader
    • 高可用用：可以解决分布式程序中的单点故障问题，每个server保存一份相同的数据副本
    • 交互中转站：可以作为不同节点交互的中转站
    • 性能惊人：ZooKeeper集群的吞吐量基准是每秒超过10000次操作
    • 服务器集群：一个leader，多个follwer；只要有超过一办的服务器还在运行，zooKeeper就能正常运行
  • 工作机制
    • 基于观察者模式，接受服务器的注册和下线，维持当前服务器列表并且与客户端交互
    • zookeeper数据模型可以看成一颗树，有统一的节点概念znode，每个znode由路径唯一标识
    • 运行过程中只要有超过一半的节点正常运行zookeeper就可以正常运行，节点分为leader和foller
  • 应用场景：
    • 统一命名
    • 统一配置
    • 统一集群管理
    • 服务器动态上下线/软负载均衡
  • 安装配置分布式zooKeeper,再次熟悉了xsync脚本
    • zoo.cfg/myid
  • 服务器选举模式：
    • conf包含服务器和客户端的启动脚本
  • 节点类型(临时？序列？)
  • 客户端shell指令
  • 监听是zookeeper最核心的功能
